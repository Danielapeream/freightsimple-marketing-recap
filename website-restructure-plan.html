01 · Executive summary

The argument in one read.

The case for restructuring

The current site contains ~240 indexable URLs across three layers: a core marketing site (22 pages), a valuable blog (37 articles), and a sprawling SEO layer (~170 pages) with massive cannibalization and thin content.

The result: 0 organic traffic and 0 ranking keywords, despite a significant indexed footprint and backlinks from high-authority domains (Crunchbase DA 91, Craft.co DA 56). Content exists. None of it earns.

This plan reduces the site from ~240 → ~88 pages (63% reduction), fixes four critical architectural problems, consolidates SEO cannibalization through 301 redirects, and establishes a scalable taxonomy that can grow intentionally in Q3+ instead of accidentally. The USA location layer is scoped to states where we have actual customer concentration — not all 50 — staying honest about real footprint (89% Canada, 11% USA).

02 · The 4 problems

What this plan actually solves.

Four architectural failures the current site has accumulated. Each one is solvable. Together, they're the reason content investment hasn't translated into organic visibility.

Problem 01

SEO cannibalization

Multiple URLs competing for the same intent. Ontario has 5 variants. BC has 6. Generic Canada has 10+. US states duplicated with and without the -ltl suffix. Google can't decide which to rank — so it ranks none.

FixOne canonical URL per location/intent. 301 redirects from all variants to the canonical.
Problem 02

Offer/content contradiction

Home states "We don't handle parcel or courier shipping" — yet dedicated pages exist for FedEx and Purolator (brands primarily associated with parcel).

FixInvestigate with Jack (Ops) whether FedEx Freight and Purolator Freight (LTL divisions) are active carriers. If yes, rewrite pages to clarify. If no, remove.
Problem 03

Broken promises

Home, FAQ, and HTML sitemap link to "Guides" and "Help Center" — neither exists in the XML sitemap.

FixCreate /guides/ as a curated hub pulling 5–6 of the best existing blog posts into pillar-page format.
Problem 04

Mixed navigation taxonomy

"Solutions" menu currently mixes three different concepts — by-problem, by-industry, by-role — creating 10+ pages that no decision-maker actually navigates through.

FixSingle adaptive /solutions/ page that changes content based on user selection (role or industry). One page to maintain, one URL for SEO concentration.

Every US state currently has two URLs (/content/{state}/ and /content/{state}-ltl/). The new approach is two-tiered:

06 · The adaptive solutions page

One URL, many audiences.

The single biggest IA shift: 11 separate pages — one per role, one per industry, one per problem — collapse into one adaptive page that swaps content based on user selection. Same SEO target, better UX, less to maintain.

┌──────────────────────────────────────────────────┐
│  HEADER: "Freight shouldn't be a headache           │
│           for [your role / your industry]"          │
├──────────────────────────────────────────────────┤
│  SELECTOR BAR:                                    │
│  I'm a:  [Ops] [Finance] [CEO] [Warehouse]       │
│  We are: [Manufacturer] [Distributor]            │
│          [Multi-location]                        │
├──────────────────────────────────────────────────┤
│  PAIN SECTION (adapts to selection)               │
│  FEATURES that matter to you (adapts)             │
│  CASE STUDY relevant to you (adapts)              │
│  CTA specific to you (adapts)                     │
└──────────────────────────────────────────────────┘

Technical approach: URL parameters drive the view, but the same HTML serves all versions. Example: /solutions/?role=finance loads the same page but JS swaps the content blocks.

SEO approach: the page itself ranks on core terms ("LTL solutions," "freight software for companies"). The variants don't need separate URLs because they serve the same search intent — differentiated by on-page UX, not by separate ranking targets.

Benefit: one authoritative page instead of 11 diluted ones.

10 · Scope

What this document is and isn't.

It is

The architectural blueprint. Defines what pages exist, how they connect, and how we migrate from old to new without losing SEO value.

It isn't

A copy document. A visual design. A build spec for engineering. Every new page needs copy written (Sprint 02), design applied (Sprint 03), and code shipped — though this is the input for all three.
